# iOS 活动编辑器未保存问题排查记录

## 背景

发布活动时，`活动说明` 和 `活动行程` 在编辑页里已经输入了内容，但返回发布表单后仍然显示占位文案：

- `点击补充活动亮点、行程和适合人群`
- `点击补充活动行程安排`

最终现象是：

- 表单底部 `提交审核` 按钮保持灰态
- 自动化流程误以为已经填完，直到提交阶段才超时失败

相关实现文件：

- `apps/velowind-app/appium/velowind_appium/modules/activity.py`
- `apps/velowind-app/appium/tests/test_activity_helpers.py`

## 这次确认过的根因

### 1. 不是“内容没输进去”，而是“退出编辑页时没真正保存”

单步调试证明：

- 在编辑页内，正文已经成功写入 `XCUIElementTypeTextView`
- 但关闭编辑页后，外层表单仍然显示占位文案

这说明问题不在 `_fill_editor_body()`，而在“收键盘 + 点击完成 + 返回表单”的交互链路。

### 2. 直接点标题坐标，真机上不一定能收起键盘

之前的思路是：

1. 输入正文
2. 点页面标题 `编辑活动说明`
3. 点底部 `完成`

但真机调试截图证明，这个固定坐标点击并没有稳定收起 iOS 键盘，`XCUIElementTypeKeyboard` 仍然存在。

### 3. “发布笔记”的编辑器关闭方式是可复用的

对照 `message_detail.py` 的笔记发布逻辑后，确认这套顺序是有效的：

1. `_hide_keyboard(driver)`
2. `_tap_outside_editor(driver)`
3. `time.sleep(0.2)`
4. 再点底部 `完成`

把这套模式用于活动说明编辑器后，真机单步验证通过：

- 键盘能收起
- 返回表单后占位文案消失
- 表单上能看到已保存内容

### 4. 原来的表单校验把这两个占位文案忽略掉了

`_required_field_markers_resolved()` 之前会忽略：

- `点击补充活动亮点、行程和适合人群`
- `点击补充活动行程安排`

这会导致：

- 即使说明/行程没保存，流程也会继续往下走
- 最终在提交阶段才暴露问题

## 最终修复模式

### 修复点 1：活动编辑器复用“发布笔记”的收键盘逻辑

在 `activity.py` 中加入和笔记编辑器同样的关闭前处理：

1. 先 `_hide_keyboard(driver)`
2. 再 `_tap_outside_editor(driver)`
3. 短暂等待键盘动画结束
4. 再点底部 `完成`

这比“直接点标题坐标”稳定。

### 修复点 2：关闭编辑器后立刻校验是否真的保存成功

在 `_fill_description()` 和 `_fill_itinerary()` 中，`_close_editor()` 之后立即检查：

- 对应占位文案是否还存在

如果占位文案还在，直接抛出断言，避免流程继续误跑。

### 修复点 3：表单必填校验不再忽略说明/行程占位文案

`_required_field_markers_resolved()` 改为检查所有未解决占位文案，确保：

- 说明没保存，流程就不能继续
- 行程没保存，流程也不能继续

## 推荐排查步骤

以后遇到类似 “iOS 编辑器里看起来输入了，但返回后没保存” 的问题，按这个顺序查：

1. 先抓编辑页截图和 XML，确认正文是否真的进了 `TextView`
2. 再抓“点击完成前”的截图和 XML，确认键盘是否还在
3. 如果键盘还在，优先复用“发布笔记”的 `_dismiss_editor_keyboard` 模式
4. 返回表单后立即检查占位文案是否消失，不要等到最终提交才判断
5. 如果表单仍灰态，先看占位文案，再看是否有其他必填项未完成

## 这次用到的关键调试结论

### 无效方案

- 固定坐标点标题 `编辑活动说明`
- 编辑器关闭后只判断“页面退回了发布页”

原因：

- 键盘不一定真的收起
- 回到发布页不等于数据已经保存

### 有效方案

- 先 `_hide_keyboard(driver)`
- 再 `_tap_outside_editor(driver)`
- 再点击底部 `完成`
- 关闭后检查占位文案是否消失

## 一句话经验

在 iOS 富文本/编辑器类页面里，`页面关闭成功` 不等于 `内容保存成功`。  
必须把“收键盘成功”“点击最终动作成功”“返回后占位文案消失”这三件事分别验证。

## 后续补充：活动行程对象化与提交流程

这份文档最初解决的是“活动说明/活动行程没保存”的问题。后续继续排查完整发布 case 时，又确认了一个独立但强相关的事实：

- `活动行程` 不是单一正文输入框
- 它实际是一个“多段行程编辑器”
- 每一段都包含 3 个字段：
  - `标题`
  - `副标题`
  - `正文`
- 新的一段不是自动出现，而是要点击右下角 `+`

### 1. 活动行程的数据结构不能再用纯文本列表

之前如果把 `activityItinerary` 当成：

- 字符串
- 字符串列表
- 或者把多天内容直接拼进一个正文

都会有两个风险：

1. 前端只填到了第一段的某一个输入框
2. 后端要求的是结构化行程，最终在提交时出现字段校验失败

后来稳定下来的方案是：

```yaml
activityItinerary:
  - title: "Day1 集合说明"
    subtitle: "石家庄集合签到"
    body: "完成签到、路线说明与安全须知确认。"
  - title: "Day2 主线骑行"
    subtitle: "峡谷耐力挑战"
    body: "完成主线路骑行并设置2个补给点。"
  - title: "Day3 返程收尾"
    subtitle: "自由骑行返程"
    body: "自由骑行返程，完成活动复盘后解散。"
```

也就是：

- `activityItinerary` 必须是 `list[object]`
- 每个对象至少明确包含 `title`、`subtitle`、`body`

### 2. 正确的活动行程填充方式

真机验证后，稳定的填充顺序是：

1. 打开 `活动行程` 编辑页
2. 第 1 段直接填 `标题 / 副标题 / 正文`
3. 在段内字段切换时，先收键盘
4. 如果还有下一段，先点击右下角 `+`
5. 等新一段出现后，再按索引填这一段的 `标题 / 副标题 / 正文`
6. 全部段落填完后，再按“编辑器关闭”流程点底部 `完成`

这里有两个关键点：

- `+` 是“新增一段行程”的唯一可靠入口
- 同一段内从 `标题` 切到 `副标题`、再切到 `正文` 时，也要考虑 iOS 键盘遮挡

### 3. 不要把“后续提交流程问题”误判成“行程没保存”

这次排查里还出现过一个很容易误判的现象：

- 页面上已经能看到行程标题
- `活动行程` 占位文案也已经消失
- `提交审核` 按钮也已经重新出现

这时说明：

- `活动行程` 本身已经填完并保存成功
- 后续如果没有提交成功，应该继续排查“提交审核之后”的阶段
- 不能再回头把问题归因到“活动行程没保存”

后面一次单独真机诊断已经明确验证过：

- `点击补充活动行程安排` 已消失
- `Day1 / Day2 / Day3` 标题都能在发布表单上看到
- 页面上已经能看到 `提交审核`

也就是说，活动行程问题和提交问题必须拆开看。

### 4. 表单回填阶段不要误触“城市名称”

这次还有一个经验值得记下来：

- 如果页面上已经回到发布表单
- `活动行程` 已保存
- `提交审核` 已可见

那脚本应该优先继续提交流程，而不是回头重新输入一次 `城市名称`

否则容易出现两类问题：

1. 无意义地重复填写字段，打断已经完成的页面状态
2. 焦点重新落入输入框，键盘再次弹起，反而挡住底部 `提交审核`

所以以后排查“为什么没提交审核”时，先看：

1. 当前页面是否已经回到发布表单
2. `提交审核` 是否已经可见
3. 是否是脚本又回头补填了不该补填的字段

### 5. 这类问题的推荐排查顺序

如果以后再遇到“活动行程填写后，最后没有提交成功”，建议按下面顺序切分问题：

1. 先确认 `活动行程` 占位文案是否消失
2. 再确认表单上是否能看到每一段行程的标题
3. 再确认 `提交审核` 是否已经出现在当前页面
4. 如果已经出现 `提交审核`，就不要继续怀疑“行程未保存”
5. 直接抓“提交审核之前”和“点击提交之后”的截图/XML，排查新的提交阶段问题

## 本次补充后的最终经验

发布活动这类 case，要把问题拆成 3 层分别验证：

1. 编辑器内容有没有真正保存
2. 结构化字段有没有按页面真实 schema 填对
3. 回到表单后，脚本有没有继续执行正确的下一步，而不是重复填旧字段

只有这样，才能避免把“提交失败”笼统归因为“前面没填成功”。
